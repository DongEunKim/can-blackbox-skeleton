# CAN Blackbox 시스템사양서

실제 운영 환경 배포를 전제로 한 시스템 요구사항 및 구성을 정의한다.

---

## 1. 개요

### 1.1 목적

CAN Blackbox는 차량 또는 산업용 CAN 버스 데이터를 수집하여 BLF(Binary Logging Format) 파일로 저장하고, AWS IoT Greengrass Stream Manager를 통해 Amazon S3로 전송하는 에지 솔루션이다.

### 1.2 적용 환경

| 구분 | 내용 |
|------|------|
| 배포 위치 | AWS IoT Greengrass Core 디바이스 (에지 디바이스) |
| 운영 모드 | 항상 실행 (백그라운드 서비스) |
| 데이터 흐름 | CAN 버스 → 로컬 BLF → Stream Manager → S3 |

---

## 2. 시스템 구성

### 2.1 구성도

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Greengrass Core 디바이스 (에지)                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌──────────┐     ┌──────────────┐     ┌─────────────┐              │
│  │ CAN 버스  │────▶│  can_logger  │────▶│  ./logs/    │              │
│  │(can0 등)  │     │  (BLF 기록)  │     │  BLF 파일   │              │
│  └──────────┘     └──────────────┘     └──────┬──────┘              │
│                                               │                      │
│                                               ▼                      │
│  ┌──────────┐     ┌──────────────┐     ┌─────────────┐              │
│  │   S3     │◀────│  uploader    │◀────│ file_watcher│              │
│  │(Cloud)   │     │(StreamManager│     │ (폴더 감시)  │              │
│  └──────────┘     │  클라이언트)  │     └─────────────┘              │
│       ▲           └──────┬───────┘                                    │
│       │                  │                                            │
│       │                  ▼                                            │
│       │           ┌──────────────────┐                                │
│       └───────────│ Stream Manager   │                                │
│                   │ (Greengrass 컴포넌트) │                            │
│                   └──────────────────┘                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 프로세스 구성

| 프로세스 | 역할 | 실행 순서 |
|----------|------|-----------|
| can_logger | CAN 수신 → BLF 저장, 로테이션 | 1 (독립) |
| uploader | 폴더 감시 → S3 업로드 태스크 전달 → 완료 후 삭제 | 2 (독립) |

두 프로세스는 독립적으로 실행되며, 별도 프로세스로 관리(예: systemd 유닛 2개)하는 것을 권장한다.

## 3. 하드웨어 요구사양

### 3.1 최소 사양

| 항목 | 최소 요구사항 |
|------|---------------|
| CPU | ARM Cortex-A 또는 x86_64 1코어 이상 |
| RAM | 512MB 이상 (Greengrass Core + 본 솔루션) |
| 저장장치 | 1GB 이상 여유 공간 (BLF 로그 임시 저장용) |
| CAN 인터페이스 | SocketCAN 지원 (can0, vcan0 등) |

### 3.2 권장 사양

| 항목 | 권장 |
|------|------|
| RAM | 1GB 이상 |
| 저장장치 | 5GB 이상 (max_total_mb 설정에 따른 충분한 여유) |
| 네트워크 | 안정적인 인터넷 연결 (S3 업로드용) |

### 3.3 CAN 하드웨어

- **실제 CAN**: CAN 트랜시버가 연결된 물리 인터페이스 (can0, can1 등)
- **테스트용**: Virtual CAN (vcan0) 지원

## 5. AWS 리소스 요구사항

### 5.1 필수 리소스

| 리소스 | 용도 |
|--------|------|
| AWS IoT Greengrass Core | 에지 디바이스에 설치 |
| Stream Manager | Greengrass 기본 컴포넌트 (활성화 필요) |
| S3 버킷 | BLF 로그 저장 대상 |

### 5.2 IAM 권한 (토큰 교환 역할)

Greengrass Core 디바이스의 토큰 교환 역할에 다음 권한 필요:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:AbortMultipartUpload",
        "s3:ListMultipartUploadParts"
      ],
      "Resource": "arn:aws:s3:::버킷명/*"
    }
  ]
}
```

### 5.3 S3 버킷

- **사전 생성 필수**: Stream Manager는 버킷을 자동 생성하지 않음
- **리전**: Greengrass Core와 동일 AWS 계정 내

## 6. 네트워크 및 인터페이스

### 6.1 CAN 인터페이스

- 프로토콜: CAN 2.0A/B
- SocketCAN 네이밍: can0, can1, vcan0 등
- 다중 인터페이스: config.ini에서 쉼표로 구분 지정

### 6.2 클라우드 연결

- S3 업로드를 위한 아웃바운드 HTTPS (443)
- Greengrass MQTT/Stream Manager: Greengrass Core가 관리

## 7. 컴포넌트 상세

### 7.1 can_logger

| 항목 | 내용 |
|------|------|
| 입력 | CAN 버스 (can0 등) |
| 출력 | BLF 파일 (./logs/CBB_YYYY-MM-DDTHHMMSS_#NNN.blf) |
| 로테이션 | 용량(rotation_max_mb) 또는 최대 로깅 시간(max_logging_minutes) 기준 |
| 재연결 | 지수 백오프 적용 |

### 7.2 file_watcher (uploader 내장)

| 항목 | 내용 |
|------|------|
| 감시 대상 | logging.output_dir |
| 감시 방식 | 폴링 (poll_interval 초) |
| 판별 | 0바이트 무시, 크기 안정 시 신규 파일로 인식 |

### 7.3 uploader

| 항목 | 내용 |
|------|------|
| 입력 | file_watcher 신규 BLF 파일 콜백 |
| 동작 | S3ExportTaskDefinition 생성 → Stream Manager append → status Success 대기 → 로컬 삭제 |
| 저장소 관리 | max_total_mb 초과 시 오래된 파일 삭제 |

### 7.4 Stream Manager (Greengrass)

| 항목 | 내용 |
|------|------|
| 역할 | S3ExportTaskDefinition 수신 → 로컬 파일 읽기 → S3 업로드 |
| S3 키 | `{s3_prefix}!{timestamp:yyyy/MM/dd}/{filename}` (날짜 파티션) |
| 완료 알림 | status stream에 Success/Failure 기록 |

## 8. 설정 사양 (config.ini)

### 8.1 [can]

| 키 | 타입 | 기본값 | 설명 |
|----|------|--------|------|
| interface | string | vcan0 | CAN 인터페이스, 쉼표 구분 다중 |
| log_interval | int | 0 | 로깅 주기(초), 0=실시간 |
| reconnect_max_retries | int | 5 | 재연결 최대 재시도 |
| reconnect_interval_sec | int | 2 | 재시도 초기 대기(초) |

### 8.2 [logging]

| 키 | 타입 | 기본값 | 설명 |
|----|------|--------|------|
| output_dir | string | ./logs | BLF 저장 경로 |
| log_prefix | string | CBB_ | 파일명 접두사 |
| rotation_max_mb | float | 10 | 로테이션 최대 용량(MB) |
| max_logging_minutes | int | 30 | 최대 로깅 시간(분), 0=무제한 |

### 8.3 [storage]

| 키 | 타입 | 기본값 | 설명 |
|----|------|--------|------|
| max_total_mb | int | 500 | 최대 총 용량(MB), 초과 시 오래된 파일 삭제 |

### 8.4 [watcher]

| 키 | 타입 | 기본값 | 설명 |
|----|------|--------|------|
| poll_interval | int | 5 | 폴더 감시 주기(초) |

### 8.5 [stream_manager]

| 키 | 타입 | 기본값 | 설명 |
|----|------|--------|------|
| use_mock | bool | true | false=실제 Greengrass Stream Manager |
| stream_name | string | CanBlackboxStream | S3 export 스트림 이름 |
| status_stream_name | string | CanBlackboxStatusStream | 업로드 완료 알림 스트림 |
| s3_bucket | string | (필수) | S3 버킷 이름 |
| s3_prefix | string | can-logs/ | S3 키 prefix (날짜 파티션 포함) |

## 9. 파일 및 경로

### 9.1 디렉터리 구조

| 경로 | 용도 |
|------|------|
| ./logs | BLF 로그 저장 (output_dir) |
| ./config.ini | 런타임 설정 |

### 9.2 BLF 파일명 형식

```
{log_prefix}{YYYY}-{MM}-{DD}T{HH}{MM}{SS}_#{NNN}.blf
예: CBB_2025-03-04T143052_#001.blf
```

### 9.3 S3 오브젝트 키 형식

```
{s3_prefix}{yyyy}/{MM}/{dd}/{filename}
예: can-logs/2025/03/04/CBB_2025-03-04T143052_#001.blf
```

## 10. 운영 환경

### 10.1 프로세스 실행

```bash
# CAN 로거 (터미널 또는 서비스 1)
python3 -m src.can_logger

# 업로더 (터미널 또는 서비스 2)
python3 -m src.uploader
```

### 10.2 systemd 유닛 예시 (권장)

```ini
# /etc/systemd/system/can-blackbox-logger.service
[Unit]
Description=CAN Blackbox Logger
After=network.target

[Service]
Type=simple
WorkingDirectory=/opt/can-blackbox
ExecStart=/usr/bin/python3 -m src.can_logger
Restart=on-failure

# /etc/systemd/system/can-blackbox-uploader.service
[Unit]
Description=CAN Blackbox Uploader
After=network.target greengrass.service

[Service]
Type=simple
WorkingDirectory=/opt/can-blackbox
ExecStart=/usr/bin/python3 -m src.uploader
Restart=on-failure
```

### 10.3 의존성

- uploader: Greengrass Core 및 Stream Manager 서비스 기동 후 실행
- can_logger: CAN 인터페이스 준비 후 실행

## 11. 데이터 흐름 및 용량

### 11.1 데이터 흐름

1. CAN 메시지 → can_logger → BLF 파일 (로컬)
2. file_watcher → 신규 BLF 감지 → uploader
3. uploader → S3ExportTaskDefinition → Stream Manager
4. Stream Manager → S3 업로드 → status stream
5. uploader → Success 확인 → 로컬 파일 삭제

### 11.2 용량 산정

| 항목 | 참고 |
|------|------|
| BLF 단위 | rotation_max_mb 기준 (예: 10MB) |
| 로컬 최대 | max_total_mb (예: 500MB) |
| S3 | 무제한 (버킷 정책에 따름) |

## 12. 보안 요구사항

### 12.1 설정 파일

- config.ini: s3_bucket 등 운영 값 포함, 접근 권한 제한 권장
- config.ini.example: 템플릿만 배포, 민감 정보 없음

### 12.2 네트워크

- S3: HTTPS, IAM 역할 기반 인증
- Greengrass: TLS, 디바이스 인증서 사용

## 13. 비기능 요구사항

### 13.1 가용성

- can_logger: CAN 끊김 시 재연결 시도
- uploader: Stream Manager 미응답 시 5분 타임아웃, 파일 유지

### 13.2 복구

- 로컬 디스크 고장 시: 미업로드 BLF 유실
- S3 업로드 실패 시: 로컬 파일 유지, storage_manager에 의해 용량 초과 시 삭제 가능
